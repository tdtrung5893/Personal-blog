Những kiến thức cần biết về Web-base

Tran Duc Trung 16/07/2018

    1. Thế nào là REST?

    - REST là viết tắt của REpresentational State Transfer, là 1 kiểu kiến trúc lập trình, định nghĩa các quy tắc trong việc thiết kế Web Service chú trọng đến các tài nguyên.

    - Các tài nguyên ở đây bao gồm các tệp văn bản, hình ảnh, video, dữ liệu động... được định danh thông qua URI và được biểu diễn bằng các file JSON, XML...

    - REST Server cung cấp quyền truy cập vào các tài nguyên, còn REST Client là nơi truy cập và chỉnh sửa các tài nguyên.

    - Các ràng buộc trong REST:

    • + Client - Server: REST hoạt động theo mô hình Client - Server, việc tách biệt này nhằm đơn giản hóa việc thực hiện kết nối và nâng cao hiệu năng cũng như việc mở rộng máy chủ
    • + Statless: Hay còn gọi là phi trạng thái, tức là cả Server lẫn Client đều không lưu trạng thái của nhau, mỗi Request từ phía Client gửi đi đều phải được đóng gói đầy đủ thông tin để Server có thể nhận
    • + Cache: Các Response có thể lấy ra từ cache, bằng cách cache Response, Server sẽ giảm tải được việc xử lý các Request và phía Client cũng nhận được thông tin nhanh hơn.
    • + Layer system: Giảm mức độ phức tạp của hệ thống, cho phép các thành phần hoạt động độc lập và từ đó giảm độ phức tạp của hệ thống để có thể dễ dàng mở rộng. Mỗi lớp chỉ sử dụng lớp dưới nó và giao tiếp với lớp ở ngay trên nó.
    • + Interface: Bằng cách tạo ra các quy ước chuẩn để giao tiếp giữa các thành phần trong hệ thống để đơn giản hóa việc Client có thể tương tác với Server. Điều này giúp cho người sử dụng hệ thống được dễ dàng hơn.

    - REST bao gồm 4 phương thức:

    • + GET Lấy dữ liệu từ trên server xuống
    • + POST Tạo dữ liệu mới trên server
    • + PUT Cập nhật dữ liệu trên server
    • + DESTROY Xóa dữ liệu trên server

    - Ưu điểm của REST:

    • + Tối ưu hơn so với 2 phương pháp thiết kế WSDL, SOAP
    • + Code ngắn gọn, dễ hiểu
    • + Dữ liệu trả về có thể hiển thị dưới nhiều định dạng: HTML, XML, JSON...
    • + Hiệu suất tốt, độ tin cậy cao, dễ phát triển và mở rộng
    • + Rõ ràng về URLs, mỗi URL đại diện cho resource chứ không phải 1 hành động

    - Nhược điểm của REST:

    • + Do phải đóng gói các thông tin trước khi gửi nên khối lượng thông tin truyền tải lớn dẫn đến tốn tài nguyên
    • + Chỉ hoạt động trên giao thức HTTP
    • + Bảo mật kém hơn so với SOAP

    2) Một Restful Resouce Controller sẽ gồm những method nào? Có những HTTP method nào để gửi request lên server? Đối với từng method thì sẽ được sử dụng bởi chức năng gì?

    - Một Resful Resource Controller sẽ gồm 7 method:

    • + Index (GET)
    • + New (GET)
    • + Create (POST)
    • + Show (GET)
    • + Edit (GET)
    • + Update (PUT/PATCH)
    • + Delete (DESTROY)

    - Có 9 HTTP method để gửi request lên server:

    • + GET: Dùng để lấy dữ liệu từ trên Server xuống
    • + HEAD: Cũng giống GET những response trả về không có body, chỉ có header
    • + POST: Gửi thông tin đến Server thông qua các form
    • + PUT: Ghi đè tất cả thông tin của đối tượng trên Server với những gì được gửi lên
    • + DELETE: Xóa thông tin của đối tượng trên Server
    • + CONNECT: Tạo 1 kết nối với Server thông qua URI
    • + OPTIONS: Mô tả các tùy chọn giao tiếp cho resource
    • + TRACE: thực hiện 1 bài test loop-back theo đường dẫn đến resource
    • + PATCH: Ghi đè các thông tin được thay đổi của đối tượng

    3) Có những HTTP response gì? Phân biệt ý nghĩa của từng loại?

    - Khi phía Client gửi lên 1 HTTP request thì Server sẽ tiếp nhận thông tin để xứ lý và trả về HTTP Respone tương ứng.

    - HTTP Response có cấu trúc tương tự như HTTP Request, chỉ khác ở chỗ HTTP Request là Request line còn HTTP Response là Status line

    Ví dụ về 1 HTTP Response

    - Status-Line bao gồm:

    • + HTTP version: Phiên bản HTTP cao nhất mà server có thể hỗ trợ
    • + Status code: Mã kết quả trả về
    • + Reason-pharse: Mô tả về status code

    - 1 số loại HTTP status code thông dụng:

    • + 1xx (Information): Server đã nhận được Request và đang tiếp tục xử lý
    • + 2xx (Success): Server đã nhận được Request và xử lý thành công
      • 200 OK: Request thành công
      • 202 Accepted: đã nhận Request nhưng không có kết quả trả về, thông báo người dùng tiếp tục đợi
      • 204 No Content: Request đã được xử lý nhưng không có thành phần nào được trả về
    • + 3xx (Redirection): Thông báo cho người dùng phải có thêm action để hoàn thành Request
    • + 4xx (Client Error): Thông báo có lỗi từ phía Client
      • 400 Bad Request: Request không đúng dạng và cú pháp
      • 401 Unauthorized: Client chưa được xác thực
      • 403 Forbidden: Client không có quyền truy cập
      • 404 Not Found: Không tìm thấy tài nguyên
    • + 5xx (Server Error): Thông báo có lỗi từ phía Server
      • 500 Internal Server Error: Có lỗi trong quá trình xử lý Server
      • 503 Service Unavailable: Server bị quá tải hoặc bị lỗi xử lý

    4) HTTPS là gì? HTTPS vs HTTP?

    - HTTPS tương tự như HTTP (Hyper Text Transfer Protocol) là 1 giao thức cơ bản dùng cho World Wide Web để truyền tải dữ liệu dưới dạng văn bản, hình ảnh, video... giữa client và server. Tuy nhiên thì giao thức HTTPS có thêm chữ S tức là Secure, có sử dụng chứng chỉ SSL (Secure Sockets Layer) giúp mã hóa dữ liệu truyền tải nhằm bảo mật giữa Server và Client

    - SSL: là 1 tiêu chuẩn bảo mật, gồm 3 lớp bảo vệ:

    • + Mã hóa: Tất cả dữ liệu được truyền tải giữa Client và Server đều được mã hóa
    • + Toàn vẹn dữ liệu: Đảm bảo dữ liệu truyền đi toàn vẹn, không thể sửa đổi hay bị hỏng mà không bị phát hiện
    • + Xác thực: Xác minh xem có thực sự bạn đang giao tiếp với Server đã định hay không

    - Cách thức hoạt động của SSL: SSL sử dụng hệ thống Public Key Infrastructure (PKI) gồm có Public Key và Private Key để mã hóa thông tin

    • + Bước 1: Client gửi yêu cầu đến Server và khởi tạo kết nối SSL với Server
    • + Bước 2: Sau khi nhận được Request, Server sẽ trả về phía Client chứng chỉ SSL và kèm theo đó là Public Key của nó.
    • + Bước 3: Sau khi nhận được SSL, Browser sẽ xác minh chứng chỉ SSL đó và tạo ra 1 key và key này sẽ được mã hóa bởi Public Key. Key được mã hóa này sẽ được sử dụng để mã hóa tất cả dữ liệu truyền tải giữa Client và Server
    • Bước 4: Khi gửi Request lên Server thì Client sẽ gửi kèm theo Key đã được mã hóa từ bước 3 và phía Server sẽ dùng Private Key để giải mã Key đó và lấy được thông tin.

    Chú ý: Key được mã hóa ở bước 3 chỉ có hiệu lực trong thời gian Session đó tồn tại.

    - So với HTTP thì HTTPS rõ ràng có tính bảo mật cao hơn hẳn vì có sử dụng mã hóa dữ liệu truyền tải.

    • + HTTP hoạt động trên mô hình Client - Server, phía Client sẽ gửi Request đến Server và chờ hồi đáp
    • + HTTPS hoạt động tương tự HTTP nhưng có bổ sung thêm chứng chỉ SSL và giao thức TSL (Transport Layer Security). Các giao thức này đảm bảo thông tin liên lạc của Client và Server luôn được bảo mật và an toàn nhất (ngăn chặn các cuộc tấn công mạng từ hacker)

    - Tuy nhiên vì không phải mã hóa dữ liệu cho nên tốc độ phản hồi của Website sử dụng HTTP nhanh hơn rất nhiều so với HTTPS. Vì vậy những trang Web đọc báo, tin tức thì nên dùng HTTP còn những trang web đòi hỏi thông tin người dùng, thanh toán trực tuyến, giao dịch ngân hàng... thì nên dùng HTTPS

    - Cổng truy cập (Port): Dùng để xác định thông tin nhận được từ phía Client sau đó phân loại và gửi đến máy chủ

    • + HTTP: Port 80
    • + HTTPS: Port 443

    5) OAuth 2 là gì? JSON Web Token là gì? OAuth 2 vs JSON Web Token?

    - OAuth: Là 1 phương thức chứng thực giúp các ứng dụng có thể chia sẻ tài nguyên với nhau mà không cần phải dùng username và password. Auth ở đây gồm có 2 nghĩa:

    • + Authentication: Xác nhận người dùng
    • + Authorization: Cấp quyền truy cập vào các Resource

    - Cách thức hoạt động của OAuth:

    • + Application yêu cầu ủy quyền để truy cập vào Resource Server thông qua User
    • + Nếu User cho phép thì Application sẽ nhận được giấy ủy quyền từ User
    • + Application sẽ gửi thông tin của định danh của mình kèm theo giấy ủy quyền đó tới Authorization Server
    • + Nếu Authorization Server xác nhận giấy ủy quyền đó hợp lệ thì sẽ trả về cho Application 1 access token
    • + Application sẽ dùng access token đó để truy cập vào Resource Server để lấy thông tin
    • + Nếu access token hợp lệ Resource server sẽ trả về thông tin mà Application đã yêu cầu

    - JSON Web Token là 1 định nghĩa về cách thức truyền tin an toàn giữa Client và Server hoặc giữa các thành viên với nhau, các thông tin trong JWT được định dạng bằng JSON, gồm có 3 thành phần là: Header, Payload, Signature

    -> cấu trúc của 1 đoạn JWT sẽ là: header.payload.signature

    • + Header: Sẽ chứa kiểu dữ liệu và thuật toán mã hóa
      •   {
          "typ": "JWT",
          "alg": "HS256"
        }
      • typ: Khai báo rằng đối tượng đã được mã hóa là JWT
      • alg: Xác định thuật toán mã hóa cho chuỗi là HS256 (HMAC SHA-256)
    • + Payload: Sẽ chứa nội dung của thông tin người dùng muốn gửi đi
      •   {
          "user_name": "admin",
          "user_id": "1513717410",
          "authorities": "ADMIN_USER",
          "jti": "474cb37f-2c9c-44e4-8f5c-1ea5e4cc4d18"
        }
      • Chú ý: jti là JWT ID
    • + Signature: Sẽ là tổng hợp của header và payload được mã hóa (ví dụ như mã hóa bằng base64UrlEncode), kèm theo đó là 1 chuỗi secret.
      •   data = base64urlEncode( header ) + "." + base64urlEncode( payload )
        signature = Hash( data, secret );

    - Cách thức hoạt động của JWT:

    • + Người dùng yêu cầu đăng nhập với username và passowrd
    • + Nếu username và passowrd hợp lệ thì Server sẽ trả cho người dùng 1 chuỗi JWT
    • + Sau đó người dùng sẽ gửi các yêu cầu kế tiếp kèm theo đó là chuỗi JWT
    • + Server sẽ nhận yêu cầu và kiểm tra chuỗi JWT, nếu hợp lệ thì thực hiện yêu cầu
    • --> JWT có thể truyền thông qua URL, qua giao thức POST hoặc là cho vào bên trong Http Header, kích thước nhỏ nên việc truyền tải thông tin cũng nhanh hơn

    Vì phần payload chứa toàn bộ thông tin cần có của người dùng cho nên không cần phải truy vấn cơ sở dữ liệu nhiều lần.

    6) Thế nào là mô hình MVC? Fat Model vs Fat Controller?

    - Mô hình MVC: là viết tắt của Model - View - Controller là 1 mẫu kiến trúc phần mềm được sử dụng trong kỹ thuật phần mềm. Nó phân bổ source code thành 3 phần, mỗi phần lại có nhiệm vụ riêng biệt và độc lập với những phần còn lại.

    • + Model: Là nơi chứa tất cả các nghiệp vụ logic, truy xuất database, đối tượng mô tả dữ liệu như các Class.
    • + View: Là nơi hiển thị giao diện đến người dùng
    • + Controller: Là nơi điểu hướng các yêu cầu từ người dùng và là nơi quản lý sự trao đổi dữ liệu

    - Cách thức hoạt động của MVC gồm có 8 bước:

    • + Bước 1: Phía Client gửi Request lên Server
    • + Bước 2: Router sẽ tiếp nhận Request và điều hướng đến action trong Controller tương ứng
    • + Bước 3: Controller xử lý Request và nếu cần nó sẽ gọi đến model
    • + Bước 4: Model sẽ truy xuất và lấy dữ liệu từ trong database ra
    • + Bước 5: Model trả lại cho Controller dữ liệu vừa lấy ra được từ database
    • + Bước 6: Controller sau khi lấy được dữ liệu sẽ đẩy tất cả về phía View
    • + Bước 7: Tại đây View sẽ render ra thành những đoạn code html thuần và trả về cho Controller
    • + Bước 8: Controller sẽ làm nốt công việc cuối cùng là trả html về Browser và hiển thị lên cho người dùng

    - FAT Model, FAT Controller: Là thuật ngữ dùng để chỉ khi ta viết quá nhiều logic tính toán hay truy vấn dữ liệu vào 1 model, 1 controller hoặc là truy vấn dữ liệu trực tiếp trong controller mà không thông qua việc đóng gói dữ liệu. Điều này sẽ gây khó khăn cho việc:

    • + Tái sử dụng
    • + Mở rộng tính năng
    • + Sửa chữa mã nguồn
    • + Bảo trì

    - Cách khắc phục:

    • + Value Object: Khuyến khích tạo ra những đối tượng nhỏ, đơn giản và cho phép bạn so sánh các đối tượng theo 1 logic nhất định hoặc đơn giản là dựa trên các thuộc tính cụ thể
    • + Form object: Là một design pattern đóng gói liên quan đến logic validating và tồn tại dữ liệu.
      • Ví dụ như khi ta có nhiều ActiveModel được update trong 1 lần submit form thì việc sử dụng Form Object có thể đóng gói tốt thực thi này
    • + Querry Object: Là 1 design pattern cho phép ta lấy các logic tính toán trong controller hay các câu truy vấn dữ liệu trong model cho vào 1 Class riêng để có thể sử dụng nhiều lần
    • + Service Object: Được tạo ra khi 1 action trong controller quá phức tạp, có trỏ tới nhiều model hay action đó tương tác với 1 service bên ngoài...

    7) Thế nào là Single Page Application? Các thư viện để làm Single Page Application nổi tiếng hiện nay?

    - Single Page Appliacation: Là 1 trang web hoặc 1 ứng dụng web mà tất cả những thao tác xử lý của trang web đều được diễn ra trên 1 trang duy nhất nhằm mục đích là nâng cao trải nghiệm người dùng tốt hơn, hiệu quả hơn và thoải mái hơn. Ví dụ như: facebook, google, github...

    - Toàn bộ resource của trang web là những file CSS, HTML, JavaScript...Dựa trên Request của người dùng SPA sẽ tiếp tục tải các HTML khác trong cùng 1 trang đó

    - So với Multiple page Application, khi người dùng yêu cầu 1 trang web, server sẽ phải tính toán và trả về toàn bộ trang web đó dưới dạng HTML. Vì vậy khi có nhiều yêu cầu được diễn ra thì quá trình tính toán sẽ mất nhiều thời gian hơn và làm chậm việc trả kết quả về cho người dùng. Ngược lại thì SPA với lần Request đầu tiên hệ thống sẽ trả về toàn bộ code xử lý cũng như code HTML của toàn bộ trang web, ở những lần request tiếp theo người dùng chỉ request những gì mình cần và server sẽ trả về dữ liệu thô, giúp rút ngắn thời gian truyền tải, nâng cao hiệu quả của người dùng.

    - 1 số thư viện dùng để làm SPA nổi tiếng hiện nay như: ReactJS, AngularJS, VueJS, EmberJS, MeteorJS...
