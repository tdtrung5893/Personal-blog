Nguyên lý SOLID

Tran Duc Trung 15/07/2018

    1) Thế nào là Design Principles?

    - Design Principles: Là tổng hợp những nguyên tắc trong thiêt kế nhằm mục đích giúp các lập trình viên tránh khỏi 1 thiết kế tồi, có 3 đặc điểm trong 1 thiết kế tồi, đó là:

    • + Cứng nhắc: Khó có thể thay đổi vì mỗi lần thay đổi sẽ ảnh hưởng tới các thành phần khác của hệ thống.
    • + Bất ổn định: Khi thực hiện 1 thay đổi có thể nó sẽ phá vỡ toàn bộ hệ thống.
    • + Kém linh hoạt: Khó có thể tái sử dụng và mở rộng khi có nhu cầu.

        2) Thế nào là SOLID?

        - SOLID: Đây là 5 nguyên tắc trong thiết kế giúp code của các lập trình viên trở nên dễ đọc, dễ hiểu, tách bạch hơn và nó còn làm cho code dễ kiểm thử, mở rộng, bảo trì sau này. SOLID là viết tắt của:

        • + Single Responsibility Principle (SRP): Nguyên tắc này nói lên rằng mỗi 1 Class thì chỉ nên giữ 1 nhiệm vụ duy nhất, tránh việc 1 class đảm nhận quá nhiều nhiệm vụ làm cho code trở nên cồng kềnh gây khó cho người đọc code và khó bảo trì sau này.
        • + Open/Closed Principle (OCP): Nguyên tắc đóng mở tức là bạn nên hạn chế việc sửa đổi source code bên trong 1 Class sẵn có và nếu muốn thêm 1 tính năng nào đó trong Class thì tốt nhất là nên viết tính năng đó trong.
        • + Liskov Subsitution Principle (LSP): Trong 1 chương trình các object của Class con có thể thay thế Class cha mà không làm mất đi tính đúng đắn của chương trình.
        • + Interface Segregation Principle (ISP): Thay vì dùng 1 Interface quá lớn và cồng kềnh ta nên tách thành những interface nhỏ hơn để tiện cho việc theo dõi và kiểm soát.
        • + Dependency Inversion Principle (DIP): Mỗi thành phần của hệ thống (Class, Module...) chỉ nên phụ thuộc vào abstraction, không nên phụ thuộc vào các Class, Module.. cấp thấp.

        3) Ví dụ về SOLID với ngôn ngữ ruby:

        - Single Responsibility Principle (SRP):

             class AuthenticateUser
               def authenticate email, password
                 if matches? email, password
                   do_authentication
                 else
                   match_error
                 end
               end
               private
                 user = find_from_db :usre, email
                 user.encrypted_password == encrypt password
               end
             end

        Trong ví dụ trên Class AuthenticateUser đã vi phạm nguyên tắc SRP vì phải xử lý 2 nhiệm vụ đó là kiểm tra match email, passowrd trong database và xác nhận user. Theo SRP ta refactor đoạn code trên như sau:

           class AuthenticateUser
             def authenticate email, password
                if Match.new(email, password).matches?
                     do_authentication
                else
                     match_error
                end
             end
           end
           
           class Match
             def initialize email, password
                @email = email
             end
             def matches?
                user = find_from_db :user, @user
                user.encrypted_password == encrypt @passwrod
             end
           end

        - Open/Closed Principle (OCP): Ta có 1 Class để quản lý sinh viên, và giờ ta có hàm payfee để phân loại học phí theo từng loại sinh viên (năm nhất, năm hai, năm ba...)

           class Student
             def initialize name, age
               @name = name
               @age = age
             end
             def profile
               puts "#{@name} is #{@age} years old"
             end
             def payfee
               if student_type == "first_year"
                 fee = standar_fee *1.1
               else if student_type == "second_year"
                 fee = standar_fee * 1.2
               else if student_type == "third_year"
                 fee = standar_fee * 1.4
               end
             end
           end

          Có thể thấy đoạn code trên vẫn hoạt động bình thường nhưng lại có điểm bất hợp lý. Nếu như sau này trường mở rộng và có thêm nhiều loại sinh viên như là sinh viên năm 4, sinh viên năm 5... thì lúc đấy code của chúng ta sẽ bị ảnh hưởng và trở nên rất cồng kềnh. Áp dụng OCP ta có thể tách các loại sinh viên ra thành nhiều Class con kế thừa class Student:

           class Student
             def initialize name, age
               @name = name
               @age = age
             end
             def profile
               puts "#{@name} is #{@age} years old"
             end
           end

           class FirstYearStudent < Student
             def payfee
               fee = standar_fee * 1.1
             end
           end

           class SecondYearStudent < Student
             def payfee
               fee = standar_fee * 1.2
             end
           end

           class ThirdYearStudent < Student
             def payfee
               fee = standar_fee * 1.4
             end
           end

        - Liskov Subsitution Principle (LSP):

           class Human
             def name
             end
           end

           class AsiaHuman < Human
             def name
               "Trung"
             end
           end

           class EuropeHuman < Human
             def name
               "Ronaldo"
             end
           end

           class Intro
             def introduce human
               puts "I am #{human.name}"
             end
           end

           a = AsiaHuman.new
           e = EuropeHuman.new
           intro = Intro.new
           intro.introduce a
           intro .introduce e
           # I am Trung
           # I am Ronaldo

        - Interface Segregation Principle (ISP): Ta có 1 interface Car được sử dụng 1 phần bởi cả 2 calss Driver và Mechanic

           class Car
             def open
             end
             def start
             end
             def change_engine
             end
           end

           class Driver
             def drive
               @car.open
               @car.start
             end
           end

           class Mechanic
             def do_stuff
               @car.change_engine
             end
           end

          Chúng ta có thể viết code tốt hơn bằng cách tách interface Car ra thành 2 interface nhỏ hơn như sau:

           class Car
             def open
             end
             def start
             end
           end

           class CarEngine
             def change_engine
             end
           end

           class Driver
             def drive
               @car.open
               @car.start
             end
           end

           class Mechanic
             def do_stuff
               @car.change_engine
             end
           end

        - Dependency Inversion Principle (DIP):

           class Report
             def body
               generate_reporty_stuff
             end
             def print
               JSONFormatter.new.format body
             end
           end

           class JSONFormatter
             def format body
             end
           end

          Với cách viết trên thì class Report sẽ bị phụ thuộc vào class JSONFormatter vì hàm print đang bị fix cứng và như thế sẽ vi phạm nguyên tắc DIP, ở đây ta có thể sử dụng nguyên tắc Dependence Injection để giải quyết vấn đề:

           class Report
             def body
               generate_reporty_stuff
             end
             def print formatter: JSONFormatter.new
               formatter.format body
             end
           end

           class JSONFormatter
             def format body
             end
           end

          Với cách viết này thì ta có thể dễ dàng thay đổi định dạng in ra của report bằng cách truyền vào tham số phù hợp

           report.Report.new
           report.print formatter: JSONFormatter.new
           report.print formatter: XMLFormatter.new
